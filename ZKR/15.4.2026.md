# Hashe

## Klasické kryptografické hashování

Kryptografické hashovací funkce (SHA-256, MD5, …) jsou navrženy tak, aby byla změna výstupu při sebemenší změně vstupu maximální — tzv. **avalanche effect** (lavinový efekt).

- Změna jediného bitu na vstupu → změní se přibližně **50 % bitů výstupu**
- Výstup (otisk) je deterministický a fixní délky
- Nelze z otisku rekonstruovat vstup (jednosměrnost)

## Perceptuální hashování

Klasické hashování je nevhodné pro porovnávání **podobnosti** obsahu — i drobná změna souboru dá úplně jiný hash.

Perceptuální hashing řeší opačný problém:
- Malá změna vstupu (jiné rozlišení, komprese, jas) → **téměř stejný otisk**
- Otisky se porovnávají **Hammingovou vzdáleností** (počet různých bitů)
- Čím menší vzdálenost, tím podobnější obsah

### Princip (obrázky/videa)

1. Vstupní médium se převede do malého, normalizovaného formátu (např. 8×8 px, grayscale)
2. Aplikuje se transformace (DCT, průměrná hodnota pixelů, …)
3. Z výsledku se sestaví binární hash (např. 64 bitů)
4. Dva soubory se porovnají: `distance = popcount(hash_A XOR hash_B)`
5. Pokud je vzdálenost pod prahem → soubory jsou si podobné

### DCT — Diskrétní kosinová transformace

- Převádí obraz z prostorové domény do **frekvenční** (podobně jako Fourierova transformace)
- Nízké frekvence = hrubá struktura obrazu (barvy, tvary) — ty jsou perceptuálně důležité
- Vysoké frekvence = detaily, šum — ty se při kompresi zahazují
- pHash vezme jen levý horní roh DCT matice (nízké frekvence) → hash je robustní vůči kompresi, změně rozlišení atd.

### Algoritmy

| Algoritmus | Zkratka | Popis |
|---|---|---|
| Average Hash | aHash | průměr pixelů |
| Perceptual Hash | pHash | DCT transformace |
| Difference Hash | dHash | gradientní rozdíly |
| Wavelet Hash | wHash | waveletová transformace |

### Použití v praxi

- **Detekce CSAM** — policie a platformy (PhotoDNA od Microsoftu) porovnávají otisky s databází zakázaného obsahu
- Detekce duplicitního / překomprovaného obsahu
- Vyhledávání podobných obrázků
- Watermarking a copyright ochrana

### Python (knihovna `imagehash`)

```python
import imagehash
from PIL import Image

img = Image.open("foto.jpg")
h = imagehash.phash(img)          # perceptuální hash

img2 = Image.open("foto_upravena.jpg")
h2 = imagehash.phash(img2)

print(h - h2)                     # Hammingova vzdálenost (0 = identické)
```

Python má i vestavěné nástroje pro základní hashing (`hashlib`), ale pro perceptuální hashování je potřeba externí knihovna (`imagehash`, `PIL`).

> Lze si vyzkoušet lokálně: `pip install imagehash Pillow`, načíst dva podobné obrázky a porovnat jejich pHash + Hammingovu vzdálenost.




## YouTube — Content ID

YouTube používá vlastní systém **Content ID**, který kombinuje perceptuální hashování s audio/video fingerprintingem:

1. Rightholder nahraje referenční soubor → YouTube z něj vytvoří **fingerprint** (audio + video samostatně)
2. Každé nově nahrané video se porovná s databází fingerprintů (~millisekudy díky ANN indexování)
3. Při shodě nad prahem → automatická akce: blokování, monetizace na účet rightholdera, nebo sledování statistik
4. Funguje i na **vystřižené/zrychlené/zrcadlově otočené** klipy — fingerprint je robustní vůči těmto transformacím
5. Audio fingerprint: podobné jako **Shazam** (spektrogram → hash vrcholů)
6. Video fingerprint: perceptuální hash klíčových snímků (keyframes) po celé délce videa

Systém není veřejně zdokumentován, ale principy jsou známé z patentů a akademických prací (např. Philips Robust Hashing, Shazam algorithm).


### Moderace obsahu (mimo copyright)

Perceptuální hashování se používá i pro **blokování zakázaného obsahu** — nejen copyright:

- **CSAM** (dětská pornografie): Microsoft PhotoDNA — hash známých materiálů je sdílen mezi platformami (Facebook, Google, YouTube, …) přes NCMEC databázi. Nový upload se porovná ještě před zveřejněním.
- **Terorismus / extremismus**: podobný princip, databáze spravuje GIFCT (Global Internet Forum to Counter Terrorism)
- Výhoda: platforma nemusí obsah "vidět" — porovnává jen otisky, což chrání i zaměstnance před expozicí

### Open source alternativy

PhotoDNA není open source (Microsoft ho licencuje zdarma platformám/neziskovkám), ale Meta vydala vlastní:

- **PDQ** — perceptuální hash pro obrázky (open source)
- **TMK+PDQF** — perceptuální hash pro videa (open source)
- Repo: `facebook/ThreatExchange` na GitHubu

## Útoky na perceptuální hashování

### Hash evasion
Útočník minimálně upraví obrázek (přidá šum, změní pár pixelů) tak, aby se hash změnil natolik, aby obešel detekci — přitom obrázek vypadá pro člověka identicky.

### Collision attack
- Výzkum z roku 2024 ukázal, že na **PhotoDNA** lze najít kolizi po ~2^14.6 operacích — extrémně málo
- Kolize = dva různé obrázky mají stejný (nebo velmi podobný) hash → nevinný uživatel může být falešně označen

### Hash inversion (rekonstrukce obsahu)
- Pomocí generativní AI lze z hashe zpětně rekonstruovat původní obrázek
- Úspěšnost: až **100 %** perceptuální podobnosti pro aHash a PhotoDNA, ~97 % pro NeuralHash a PDQ
- Ohrožuje soukromí uživatelů

## Apple NeuralHash — kontroverze (2021)

- Apple oznámil skenování CSAM **přímo na zařízení** uživatele (před uploadem do iCloudu)
- Výzkumníci během několika dnů nalezli kolize NeuralHashe — systém šlo snadno obelstít
- Masivní kritika z pohledu soukromí → Apple projekt nakonec zrušil

## Novinky

- **DINOHash** (2025) — aktuální state-of-the-art, robustnější vůči útokům než NeuralHash nebo klasické DCT schémy; využívá self-supervised DINOv2 features
- **Stable Signature** (Meta, 2023) — neviditelný watermark zabudovaný přímo do latentního prostoru difuzních modelů; hybridní přístup watermark + perceptuální hash pro detekci AI-generovaného obsahu

# Kryptografické hashovací funkce

## Merkle-Damgård konstrukce

Základ většiny klasických hashovacích funkcí (MD5, SHA-1, SHA-256).

**Princip:**
1. Vstupní zpráva se rozdělí na **bloky** pevné délky
2. Přidá se **padding** + délka zprávy na konec
3. Každý blok se zpracuje **kompresní funkcí** spolu s výstupem předchozího bloku
4. Na začátku stojí **IV** (inicializační vektor), na konci je finální hash

```
IV → [blok 1] → [blok 2] → [blok 3] → hash
         ↑           ↑           ↑
     kompresní   kompresní   kompresní
      funkce      funkce      funkce
```

**Slabina — Length extension attack:**
- Pokud znáš `hash(zpráva)`, lze bez znalosti původní zprávy dopočítat `hash(zpráva + padding + cokoliv)`
- Zranitelné: MD5, SHA-1, SHA-256
- Nezranitelné: SHA-3 (Keccak používá jiný princip — sponge construction)



**Slabina — navázání na vnitřní stav:**
- Na vnitřní blok Merkle-Damgård jde navázat (length extension attack)
- Řešení: přidání **soli** (salt), více iterací (rounds), nebo přepnutí na jiný design
- Merkle-Damgård konstrukce je dobrá, ale implementace nad ní musí tyto slabiny ošetřit

## Typy hashovacích funkcí dnes

V praxi narazíme na různé přístupy:

- **Merkle-Damgård** — MD5, SHA-1, SHA-256 (klasika, ale slabiny popsány výše)
- **Sponge construction** — SHA-3 / Keccak (žádný length extension attack)
- **Stromové struktury** — **Merkle trees**

## Merkle trees (Merkleovy stromy)

Stromová struktura kde každý uzel obsahuje hash svých potomků:

```
        Root hash
       /          \
   H(A+B)        H(C+D)
   /    \         /    \
 H(A)  H(B)   H(C)  H(D)
  |     |      |     |
  A     B      C     D
```

- Změna jediného listu → změní se hash celé větve až do rootu
- Umožňuje **efektivní ověření** jednoho záznamu bez stažení celého datasetu (Merkle proof)
- Použití: **Bitcoin** (transakce v bloku), **Git** (stromy commitů), **IPFS**, **Certificate Transparency**
